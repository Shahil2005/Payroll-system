Application Overview

Croar Payroll

A multi-tenant payroll management platform — define salary structures, run payroll cycles, and issue payslips through a guarded, permission-controlled lifecycle.

Stack: FastAPI (async)  ·  Next.js + Tailwind  ·  PostgreSQL

Date: 17 June 2026

1. Executive Summary

Croar Payroll is a decoupled single-page application backed by a JSON API. The Next.js client handles presentation; the FastAPI backend owns every business rule, calculation, and persistence concern. The system is multi-tenant — every record is scoped to a company — and protected by JWT authentication with role-based access control.

Its correctness-critical core is mature: array-based salary structures, a deterministic Decimal calculation engine, statutory snapshots, an audited approval workflow, and payslips that are released only when a cycle is fully paid. Recent work has extended it toward a managed-service feature set: per-run adjustments, CSV/PDF reports, an audit trail, an income-tax (TDS) engine, and an org-settings module.

Multi-tenant by design

Every query is scoped to the signed-in user's company. Tenants only ever see their own data.

Secure by default

JWT auth + fine-grained payroll:* permissions enforced on the API and mirrored in the UI.

Auditable

Append-only audit log records who did what and when; activity is browsable in-app.

Correct money math

Pure Decimal engine, rounded per line (ROUND_HALF_UP) to avoid cent drift.

3. Architecture

A decoupled SPA + API. The Next.js client talks to FastAPI over JSON; the backend is layered so each tier has a single responsibility.

┌─────────────────────────────┐ ┌──────────────────────────────────────┐ │ Next.js (3000) │ HTTPS │ FastAPI (8000) │ │ App Router pages │ ◄─────► │ Routers → Services → Models │ │ AuthProvider (JWT in store) │ JSON │ JWT auth + RBAC dependencies │ │ utils/api.ts apiClient │ │ async SQLAlchemy / asyncpg │ └─────────────────────────────┘ └───────────────────┬──────────────────┘ ┌──────────────┴───────────────┐ │ PostgreSQL │ Redis (jobs) │ └───────────────────────────────┘

Backend layering

  • Routers stay thin — validate input, declare the required permission, delegate.
  • Dependencies handle get_current_user, company scoping, and require_permission.
  • Services hold testable business logic — e.g. compute_payslip is pure math with no DB dependency.
  • Models are SQLAlchemy ORM with constraints and soft-delete columns.

5. Payroll Lifecycle

A cycle moves through a guarded state machine. Each transition requires a specific permission and is rejected (409) from any other state.

run (payroll:run) approve (payroll:approve) mark-paid (payroll:pay) ┌───────┐ ──────────► ┌────────────┐ ──────────► ┌──────────┐ ──────────► ┌──────┐ │ DRAFT │ │ PROCESSING │ │ APPROVED │ │ PAID │ └───────┘ ◄─ re-run ─ └────────────┘ └──────────┘ └──────┘ │ │ ▲ └──── cancel (payroll:manage) ────► ┌───────────┐ payslips released│ │ CANCELLED │ (only at PAID) ──┘ └───────────┘
  • Run is allowed on DRAFT or PROCESSING (re-run recalculates idempotently in place).
  • Cancel is allowed from any non-PAID state.
  • A PAID cycle is terminal — it cannot be re-run, cancelled, or deleted.
  • Payslips are visible only once the cycle is PAID.

6. Roles & Permissions

Permissions are fine-grained payroll:* capabilities; roles bundle them. The API enforces them via a require_permission(...) dependency (403 on failure); the frontend hides disallowed nav and blocks direct navigation.

7. Payslip Calculation

compute_payslip (in app/services/payroll_service.py) is pure, deterministic Decimal math:

  • Resolve earnings — each line is fixed, a percent of another line's code, or a percent of gross-so-far when percent_of is omitted.
  • Apply LOP proration — multiplier = (working_days − lop_days) / working_days on a fixed 30-day basis; earnings are pro-rated.
  • Resolve deductions — same line rules, evaluated against pro-rated earnings; statutory + estimated TDS where enabled.
  • Net pay = gross − total deductions.
Every resolved line is rounded to 2 dp (ROUND_HALF_UP) before summing to avoid cent drift. LOP greater than the working-days basis reports that employee in the run's skipped list rather than failing the whole run.

8. Feature Highlights

Employees

Full CRUD, unique email per company, soft delete that preserves payslip history and deactivates structures.

Salary Structures

One active per employee, array-based earnings/deductions, LOP, live monthly estimate while editing (server-computed).

Payroll Cycles

Idempotent runs, employees without a structure surfaced in a skipped list, per-cycle roll-up totals.

Per-run Adjustments

One-time bonus / arrears / deductions applied on a cycle, editable only in DRAFT/PROCESSING.

Reports & Audit

CSV + PDF salary-register and payroll-summary exports; append-only audit trail with an in-app activity page.

Taxes & TDS

Employee IT declarations (both regimes), an opt-in income-tax estimator, and TDS challan records.

Org Settings

Editable organisation profile (legal name, PAN/TAN, address) — read for all, write for admins.

Dashboard

Company-wide overview: coverage, structure counts, cycle status, disbursed vs. pending — one server call.
